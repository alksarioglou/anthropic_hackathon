## Project Overview

We are building an **Agentic AI system** for the **Swiss Life Claude Builders Hackathon** that transforms a rough software idea into a complete, implementation-ready software plan. The system guides users through the full SDLC workflow:

**Idea вҶ’ Vision вҶ’ Requirements вҶ’ Architecture вҶ’ Backlog вҶ’ Tests**

The workflow supports **iteration and retro-feedback** between all stages вҖ” changes at any stage propagate backward and forward to keep all artifacts aligned, non-redundant, and correct.

## Tech Stack & Codebase Architecture

This is a **Next.js + Convex** full-stack app:

- **`app/`** вҖ” Next.js App Router. `layout.tsx` wraps all pages; `page.tsx` is the root route. Add new routes as `app/[route]/page.tsx`.
- **`convex/`** вҖ” Convex backend functions (queries, mutations, actions) and schema. The `_generated/` subdirectory is auto-generated by `npx convex dev` вҖ” never edit it manually. To add backend logic, create `.ts` files directly in `convex/` and define functions using `query`, `mutation`, or `action` from `convex/server`.
- **`types/`** вҖ” Shared TypeScript types for the frontend.

### Key Details

- **Path alias**: `@/*` maps to the project root (e.g., `@/convex/...`, `@/types/...`).
- **Styling**: Tailwind CSS v4 (PostCSS-based, no `tailwind.config` needed).
- **No Convex schema yet**: `convex/schema.ts` does not exist. Adding one will improve type safety вҖ” run `npx convex dev` after creating it to regenerate `_generated/dataModel.d.ts`.
- **Convex client setup**: The Convex provider (`ConvexProvider` from `convex/react`) needs to be added to `app/layout.tsx` before using `useQuery`/`useMutation` hooks in client components.
- **Package manager**: pnpm (use `pnpm` for all installs).

### Stage 1 вҖ” Onboarding (Owner: ALK)

The entry point. An hourglass/loading screen leads into a guided setup flow that collects everything the downstream agents need:

1. **Setup Questions** вҖ” The onboarding agent asks the user:
   - Is this an internal (on-premise) or external (cloud) project?
   - What dashboard style: Business or Technical?
   - What AI prompting mode to use?
2. **UI Selection** вҖ” User picks their preferred interface:
   - Dashboard view
   - Hanette (chat) view
   - Direct prompt-based interaction
3. **Add Files** вҖ” User uploads relevant context documents (briefs, specs, existing docs)

The onboarding agent packages all of this into a structured context payload and passes it downstream to Stage 2.

### Stage 2 вҖ” Business Specs вҶ’ Tech Specs Generation (Owner: Jannis)

The core generation pipeline, broken into two sequential sub-stages:

**Sub-stage 2a: Business Specs** вҖ” From the onboarding payload, agents generate:
- **Business Dashboard** with panels for: Vision, Requirements, Cost (estimated effort/complexity), Competitive Analysis (external mode)
- These artifacts are human-readable and oriented toward business stakeholders

**Sub-stage 2b: Tech Specs** вҖ” Business specs feed directly into technical generation:
- **SDLC Spec Artifacts**: Vision, Requirements, Architecture, Frameworks, Tests, Backlog
- **Technical Dashboard** with panels for: Tests (generated test definitions), Plans (architecture and backlog), PRs (suggested implementation structure)
- PR-style review flow where the user can approve or request changes to generated artifacts

### Stage 3 вҖ” Feedback & Retro-Alignment Loop (Owner: Damiano)

The glue that makes the system truly agentic. Damiano's feedback agent:

- Watches for **any human refinement** at any stage (e.g., "Add role-based access", "Only high-risk changes need two approvals")
- **Propagates changes bidirectionally** вҖ” back into Business Specs and forward into Tech Specs
- Cross-checks all artifacts for **redundancies, conflicts, and gaps**
- Presents a **diff/PR-style view** of what changed across stages
- Asks for user approval before committing the re-aligned artifacts
- Triggers re-generation in Stage 2 as needed, creating a continuous improvement cycle

## How Agentic AI Is Used

The system uses Claude as the core AI engine across every SDLC stage:

1. **Idea Intake Agent** вҖ” Parses the user's initial idea (text + uploaded files) and produces a structured seed for all downstream stages
2. **Vision Agent** вҖ” Generates a clear product vision statement from the idea
3. **Requirements Agent** вҖ” Extracts and refines functional and non-functional requirements
4. **Architecture Agent** вҖ” Proposes system architecture, technology choices, and component design
5. **Backlog Agent** вҖ” Breaks requirements into actionable backlog items (epics, stories, tasks)
6. **Test Agent** вҖ” Generates test definitions and acceptance criteria for each requirement/story
7. **Alignment Agent** вҖ” Cross-checks all artifacts for redundancies, conflicts, and gaps; re-aligns when any stage changes
8. **Competitive Analysis Agent** (external mode) вҖ” Researches similar tools/products and feeds insights back into requirements

### Retro-Feedback Loop
When a user refines any artifact (e.g., "Add role-based access"), the system:
- Identifies which downstream and upstream artifacts are affected
- Re-generates or updates those artifacts
- Presents a diff/PR-style view of what changed
- Asks for user approval before committing changes

## Key Principles

- **Always keep artifacts aligned** вҖ” no redundancies, no conflicts across SDLC stages
- **Human-in-the-loop** вҖ” AI generates, humans review and refine
- **Two interfaces, one engine** вҖ” Business chat bot and Technical dashboard both powered by the same agentic pipeline
- **Iterative by design** вҖ” every stage can trigger updates in every other stage
- **Demo-ready** вҖ” use the sample use cases to show a compelling end-to-end flow
